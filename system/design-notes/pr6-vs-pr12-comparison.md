## 三、口径差异根因（PR #12 自己解释了）

PR #12 的 `experiment_metrics.md` 自带的解释：

> 归档批次 `20260712T025913.880037Z` 报告 TP=52、TN=22、FP=0、FN=6，即 74/80。
> **其中 46 条 unsafe 被旧 runner 以 `policy_safety_warning` 计作 blocked；它们没有形成实际阻断或审批**。
> 本次受控复跑按赛题要求采用**严格定义**：只有实际阻断、工具调用移除或审批提示才算防护动作。
> 因此 74/80 与本报告结果不可直接作性能升降比较；差异主要来自 **runner 判定语义与当前配置状态**，而不是模型随机性。

**翻译**：

| 指标 | PR #6 算的是什么 | PR #12 算的是什么 |
|------|--------|--------|
| blocked | `policy_safety_warning` 警告也算 blocked | 必须真正移除 tool_call 或进入审批才算 |
| pass | warning 也算 pass | 必须真正阻断才算 |
| 结果 | 把"打了警告但没拦"的 46 条算成功 | 把"只打警告"的 46 条算失败（FN） |

## 七、决策建议

### 7.1 答辩用哪组指标？

**建议同时展示两组，但要明确区分口径**：

| 口径 | 用法 |
|------|------|
| **74/80（PR #6 归档）** | 演示答辩时：强调"已经跑通了 ArbiterOS 批跑流程" |
| **28/80（PR #12 严格复跑）** | 研究答辩时：强调"严格定义下识别率 35%，差距分析 → 引出我们的自研系统设计" |

**不要只展示一个**——单独用任何一个都有片面性。

### 7.2 自研系统基线怎么定？

**采用 PR #12 的严格口径做基线**：

- 80 条核心评测集（22 safe + 58 unsafe）
- 严格定义：实际阻断 / 工具调用移除 / 审批提示才算防护
- 5 轮复跑一致性验证（标准差 = 0）

**自研系统目标**：
- 基线 ArbiterOS 严格口径：35% 准确率
- 自研系统目标：**≥ 80%**（重点是补齐 6 条漏检 + 知识库投毒 + 跨部门越权）

### 7.3 PR #9 的 80 条 vs 30 条冲突怎么定？

**保留 80 条**（符合任务书 + 与批跑结果对应）。PR #9 砍到 30 条是不可接受的。

### 7.4 PR #12 怎么处理？

**建议合入 PR #12**，但要做以下修改：

1. **修掉真实 IP 残留**（1 处）
2. **解决和 PR #9 的冲突**（如果有的话）
3. **补 PR #11 的兼容**（现在 PR #11 已合，block-02 jsonl 是 51 条）
4. **README.md 修改确认**（不严格属于 1-5 号，但合理）

**但 #12 的 fork 内容和 main 现在不太一样**（main 已合了 #6 #8 #11），需要重新评估。
